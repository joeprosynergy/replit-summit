❌ What Is Architecturally Wrong

At least ONE of these is still happening in useSectionContent.ts:
	•	CMS-first mode activates after fetch completion instead of fetch success
	•	CMS data is merged even when HTTP status is 401 / 403 / 404
	•	Bare filenames ("greenhouse-1.jpg") are allowed to overwrite valid defaults
	•	CMS arrays replace default arrays wholesale instead of being validated element-by-element

Any of the above violates CMS-first single-source-of-truth rules.

⸻

🎯 Required Fix (Strict, Non-Negotiable)

1️⃣ CMS-First Activation Rule (MANDATORY)

CMS-first mode must ONLY activate when:
	•	Fetch succeeds (200)
	•	Data is present
	•	Data passes validation

If CMS returns 401 / 403 / 404 / error:
	•	CMS-first = false
	•	Defaults remain authoritative
	•	CMS content is ignored completely

There must be no merge path after a failed CMS fetch.

⸻

2️⃣ Image Validation at Merge Layer (MANDATORY)

At the single merge layer (useSectionContent.ts):
	•	CMS image values MUST be validated before merge
	•	Invalid values MUST be ignored

Reject:
	•	"greenhouse-1.jpg"
	•	"image.png"
	•	""
	•	null
	•	undefined

Accept ONLY:
	•	Absolute URLs (https://…)
	•	Root-absolute paths (/assets/...)
	•	Data URLs (data:image/...)

Invalid CMS values must never overwrite defaults.

⸻

3️⃣ Array Merge Rules (MANDATORY)

CMS arrays (e.g. image galleries):
	•	Must be merged index-by-index
	•	CMS elements override defaults only if valid
	•	Invalid CMS elements are skipped
	•	Defaults remain intact when CMS array is shorter or invalid

Under no circumstances may this occur:

images: cms.images


⸻

4️⃣ Enforce “CMS Unavailable” Short-Circuit

If CMS endpoints fail:
	•	Log "CMS unavailable – using defaults"
	•	Do NOT mark CMS as resolved
	•	Do NOT update CMS-based state
	•	Do NOT trigger CMS-first mode

CMS failure ≠ CMS resolution.
